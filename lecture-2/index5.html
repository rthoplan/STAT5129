1NF: No Repeating Groups

The rule: Each cell in your table must hold a single, atomic value. You can't have lists or sets of values stuffed into one cell.

Anti-Pattern: Repeating Group

Here, the temps column stores multiple temperature readings separated by semicolons. This is hard to query (e.g., "find the average temperature").

2NF: No Partial Dependencies

The rule: This applies to tables with a composite primary key (a key made of two or more columns). It states that all non-key columns must depend on the entire composite key, not just a part of it.

Anti-Pattern: Partial Dependency

In this table, the primary key is (day_id, plant_id). However, plant_name only depends on plant_id, not on the day. This is a partial dependency and causes data redundancy.

3NF: No Transitive Dependencies

The rule: No non-key attribute should depend on another non-key attribute. In other words, all attributes must depend only on the primary key.

Anti-Pattern: Transitive Dependency

In this table, reading_id is the key. The station_region depends on the station_id, which in turn depends on the key. This is a transitive dependency: reading_id → station_id → station_region.

Why Normalise? To Avoid Anomalies.

Normalisation prevents common database problems by ensuring data is organized logically. This keeps your data accurate, consistent, and efficient.

1. Avoid Update Anomalies

Problem: If a plant's name changes, you'd have to find and update every single record for that plant. Miss one, and your data is inconsistent.
With normalisation: You only update the name in one place in the `Plants` table.

2. Avoid Insertion Anomalies

Problem: In the unnormalised `Power Generation` table (from 2NF), you can't add a new plant until it has generated power, because `day_id` is part of the key and cannot be null.
With normalisation: You can add a new plant to the `Plants` table at any time.

3. Avoid Deletion Anomalies

Problem: If you delete the last power generation record for "Wind Turbine B", you also lose the fact that a plant with that name ever existed.
With normalisation: Deleting from the `Power Generation` table doesn't affect the plant's record in the `Plants` table.

Other Key Benefits

  • Minimal Storage: Eliminates redundant data, saving space.
  • Cleaner Joins: Data is structured logically, making queries (which use joins) more straightforward and less error-prone.
