An interactive guide to understanding relational databases.

First, What is a Relational Database?

Before we talk about the management system (RDBMS), let's understand the database itself.

Imagine a collection of organised spreadsheets (like Excel) that are all linked to each other. In a relational database:

  • Each spreadsheet is called a table.
  • Each row in a table is a record (e.g., a specific customer).
  • Each column is an attribute (e.g., customer name or phone number).

For example, a Mauritian hotel group might have a 'Customers' table and a 'Bookings' table. Instead of repeating a customer's name and address for every booking they make, the 'Bookings' table simply links to the customer's unique ID in the 'Customers' table. This method of storing data in separate but linked tables is the core idea of a relational database. It keeps data organised, efficient, and prevents errors.

Users & Applications

This is the starting point. A user or an application (like a website or mobile app) needs to interact with data. They don't access the data directly; instead, they formulate a request to retrieve, add, update, or delete information.

SQL Queries

The request is translated into a SQL (Structured Query Language) query. SQL is the universal language for communicating with relational databases. For example, SELECT * FROM reservations WHERE hotel_location = 'Flic-en-Flac'; is a query to get all hotel reservations in Flic-en-Flac.

RDBMS Engine

The RDBMS is the software that receives and processes the SQL query. It's the "brain" of the operation. It parses the query, checks for permissions, creates an optimal execution plan, and then performs the requested action on the database.

Data Tables

The data itself is stored in structured tables, which consist of rows and columns. Each table represents an entity (like 'Customers' or 'Products'), and relationships can be defined between them. The RDBMS engine reads from or writes to these tables to fulfill the query.

The ACID Guarantee

An RDBMS ensures data reliability through four key properties, collectively known as ACID. This is a contract that guarantees transactions are processed reliably.

Click a property to see its description.

Atomicity

Ensures a transaction is an "all or nothing" operation. If one part fails, the entire transaction fails. Think of booking a flight and hotel package on Air Mauritius. The flight must be booked AND the hotel reserved. If the hotel booking fails, the flight booking must be automatically cancelled too.

Consistency

Guarantees that a transaction will only bring the database from one valid state to another. Any data written to the database must be valid according to all defined rules. For example, a National ID number must follow the correct format.

Isolation

Ensures that concurrent transactions do not interfere with each other. Imagine the last ticket for a concert at the Caudan Arts Centre is being sold online. Two customers click 'buy' at the same moment. Isolation guarantees that one customer's entire transaction (reserving the ticket, processing payment) is completed before the other customer's transaction begins. This prevents both people from accidentally buying the same single ticket.

Durability

Guarantees that once a transaction has been committed, it will remain so even if there's a power outage or system crash. Your Juice payment confirmation is permanent and won't be lost.

In The Real World

Popular RDBMS Software

These systems are used everywhere, from powering the online banking portal of MCB to managing student records at the University of Mauritius via its Oracle Enterprise Resource Planning (ERP) system.

Why does it matter?

Without an RDBMS, managing large datasets would be chaotic. Data could become inconsistent, duplicated, or lost. It would be difficult to enforce rules, ensure data integrity, and prevent errors, especially with multiple users accessing the data simultaneously.

Popular RDBMS: A Quick Comparison

MySQL

The world's most popular open-source database. Think of it as a reliable, high-performance car that's great for most everyday driving.

  • Main Advantage: Speed and Simplicity. It's incredibly fast for reading data and is relatively easy to set up and manage, making it a top choice for web applications.
  • Key Difference: Popularity. Its massive community provides endless online resources, making it easy to find help and tutorials.

PostgreSQL

The most advanced open-source database. If MySQL is a great family car, PostgreSQL is a powerful, all-terrain vehicle built for complex jobs.

  • Main Advantage: Advanced Features. It's better at handling complex queries and large datasets, making it ideal for financial and scientific applications.
  • Key Difference: Extensibility and Compliance. You can define your own data types, and it strictly follows SQL standards for high data reliability.

Oracle

The industrial-grade, armoured truck of databases, built for the largest and most critical corporate applications.

  • Main Advantage: Enterprise Power. Designed for massive-scale systems that demand the highest level of performance, security, and availability, like those in major banks.
  • Key Difference: Official Support and Cost. As a commercial product, it comes with dedicated professional support but also has significant licensing costs.
